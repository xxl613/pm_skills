# 04 · 交互与文案规范

> 这是「可变层」的判断依据：给规则与语气，不给死值。落地时用 `components/` 的具体组件。

## 交互规范

### 增删改查（CRUD）
- **新建/编辑**：优先用 **C 抽屉**（右侧表单）承载，对话不打断；轻量确认用 dialog。表单控件用 `form.html`。
- **删除/不可逆操作**：必须 **dialog 二次确认**。确认框文案点明影响范围与对象名；主按钮用**危险样式**（`--v5-color-error`），次按钮「取消」在左。
- **查看详情**：作为对话流里的**详情卡片**（card），或 C 抽屉内容；不另开独立详情页原型。

### 确认 / 取消
- 弹窗按钮**次（取消）在左、主（确定）在右**，整组右对齐。
- 主按钮文案用**动词短语**描述结果（「删除作业」「保存修改」「发布助教」），不用泛化的「确定/提交」当主操作（确认框可用「确定」）。
- 取消 = 不产生副作用、关闭即可；危险操作的主按钮必须与「取消」视觉可区分。

### 三态：加载 / 错误 / 空
- **加载**：轻量状态反馈（骨架/loading），动效克制并尊重 reduced motion；长任务给进度（如抽屉执行计划的进度条）。
- **错误**：用 toast(error) 或表单内联错误；**文案给下一步**（「上传失败，请检查网络后重试」），不只报「出错了」。
- **空**：用 `empty-state`（图标+说明+可选主操作），说明为什么空 + 引导第一步。
- **不靠颜色单独表达状态**：语义色必须配图标或文字（无障碍）。

### 反馈层级
| 轻重 | 用 |
|---|---|
| 轻量、自动消失 | toast |
| 需用户确认/决策 | dialog |
| 持续旁路信息/表单 | C 抽屉 |
| 内容承载 | card |

## 文案规范（品牌气质：专业 / 克制 / 清晰）

像可信赖的教育 SaaS 工作台。帮老师快速理解**当前对象、下一步动作、影响范围**。

### 要
- 按钮用**动词短语**，结果导向。
- 错误/空状态给**下一步**。
- 简洁可扫读；术语与产品一致（V5 小管家、课代表 Agent、普通 Agent）。
- 数字/状态前置，便于一屏判断。

### 不要
- 营销腔、夸张词（「强大」「智能」「一键搞定」）。
- 把 PRD 说明、路由名、评审备注混进用户可见区。
- **把演示/导览/自指脚手架混进真实页面**：「查看功能演示」「立即体验」「进入XX对话」——真实用户已在该功能内，产品上线后不会有这些。判据：**"上线后真实用户在这个页面会看到它吗？"** 不会就删。
- 泛化科技黑话堆砌「AI 感」——AI 感靠上下文、预填、能力关系表达。
- 模糊归因（「系统认为…」），含糊的「可能/也许」。

### 好 / 坏 示例
| 场景 | ✅ 好 | ❌ 坏 |
|---|---|---|
| 删除确认主按钮 | 删除作业 | 确定 |
| 上传失败 | 上传失败，请检查网络后重试 | 出错了 |
| 空作业列表 | 还没有作业，去创建第一个吧 | 暂无数据 |
| 发布 CTA | 发布助教 | 立即体验强大的 AI 助教！ |
| 学业规划助手页内 | 直接就是可用的对话输入台 | 「进入学业规划助手对话」按钮 |
| 功能页 | 功能本身即用即见 | 「查看功能演示」入口 |
