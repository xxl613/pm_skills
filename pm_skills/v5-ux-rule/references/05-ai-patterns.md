## 对话生命周期（任务型对话页的固定顺序）

```
用户请求 → 主 Agent 接收/判断 → 派发 → 场景 Agent 接收/生成 → 产物 → 完成句 → 后续建议 → 输入台恢复
```
- **完成句不得早于产物**；场景 Agent 不得在主 Agent 派发前出现；后续建议不是完成结果。
- **Agent 角色**：主 Agent 接收/理解/路由/总结/交还控制权；场景 Agent 执行具体任务、产出产物。
- **过程链装在「过程卡」里（Figma 实测）**：派发→专员处理这段过程消息统一放在一个白色圆角卡容器内（白底 · 0.5px `--v5-color-border` 描边 · 圆角 16），**每个 Agent 段落右缘配一张该 Agent 专属的 3D 角色插画**（178×112，`assets/media/illustration-desk-*.png`，"数字员工"隐喻）；**最终回答由主 Agent 输出，普通消息形态——无卡容器、无插画**，消息头带时间戳。
- **完成后过程卡默认收起，可展开回看**：任务完成后对话流里只保留最终回答，过程卡收起（详细过程也可在「执行计划」抽屉的运行日志回看）；用户点开时展开全链。样例页 `chat-flow-full.html` 画的是展开态。
- 用**头像 + 职能化名称**区分 Agent：名称本身要能表达职能（如「学业规划助手」「教学活动专员」），**不需要再加一段描述性角色副标题**（如"从入学到授位的培养导航员"这类文案，已废弃）；名称右侧只跟一个可选的过程状态标签（见下）。不靠名字单点区分时才用头像形状/配色兜底。**一条消息一个说话人**，不在同一消息块混 Agent 声音。任务完成后交还主 Agent，让用户知道下一句谁接。
- **头像规格（Figma 实测）**：36×36 圆角 10，真实角色 PNG 配每 Agent 专属底色（小管家 `#7b859f`、其余按项目分配 `#979f7b`/`#927b9f`/`#b16aba`/`#ffb84d`/`#6fb7e9`/`#4e9e96` 等柔和色）。**头像来源 = `assets/avatars/` 形象备选库（35 个，含历史人物 IP），选用规则见 SKILL.md `avatars/` 条目；过程卡插画来源 = `assets/media/illustration-*`（15 张办公形象）。均禁止库外图。**
- **流式渐进置灰**：场景 Agent 逐段生成时，**较早段落变 `--v5-color-text-secondary`，只有最新段落用正文色**，用户视线自然锚定最新内容；展开回看态（非流式）则全部正文色。

## 思考 / 工具调用 / 交接

- 思考态只展示用户可理解的状态：`已接收` / `思考中` / `调用能力` / `派发中`。**绝不暴露内部 workflow 节点、原始系统日志、推理链**。
- **过程状态挂在消息头，不单独占一条消息**：`已接收`/`思考中`/`调用能力`/`生成中`这类过程态，用消息头（头像+名称+时间那一行）里追加的一个小标签表达，不再单独起一条 pill 消息打断阅读顺序。
  - **消息头结构**（从左到右，各段用细竖线 `|` 分隔，不是空格）：`[头像][职能化名称] | [时间，可选] | [⚡ 状态标签，可选]`。
  - **字号颜色统一（Figma 实测修正）**：名称/时间/状态标签三段**同用** 14px、**同用 `--v5-color-text-secondary`（#86909c）**，名称**不加粗、不加深**——整行消息头都是弱化的辅助信息，层级靠正文与消息头的对比形成，不在头内再分层。（旧口径"名称 semibold 深色"已废弃。）
  - **状态标签格式**：`⚡`（真实图标 `assets/icons/msg-skill-bolt.svg`，16px）+ 内容，两种写法都可以——`{技能/任务名} {状态词}`（如 `作业完成度分析 进行中…`）或 `调用技能：{技能名}`（如 `调用技能：学情画像分析`）。
  - **生命周期**：只在该消息仍在处理/刚触发技能调用时出现；产物展示后要去掉，或换成简短的完成态措辞，不能让"进行中…"和结果正文同时停留在同一条消息里。
- **交接用「拍了拍」mention 条，取代旧的居中弱 pill**：派发方（主 Agent 或正在拆分子任务的场景 Agent）先以**完整消息**说明"收到请求→思考该交给谁→决定交给谁"（两步走，具体措辞每页自定，不强制模板，但语义顺序固定：先思考、后指派），消息正文最后接一行「拍了拍」条：`@派发方 拍了拍 @接收方 说，{一句开场白}`。
  - **不是独立居中元素**：这行拍了拍条是派发方那条消息的**最后一个子元素**（在 `.v5-msg-ai__body` 内，紧跟正文），不再单独起一条无头像的居中 pill。
  - **两个 mention 用 chip 样式（Figma 实测）**：chip 统一为高 30、圆角 4、`--v5-color-bg`（#f7f8fa）底、14px **semibold**；`@派发方` 字色 `--v5-color-text-secondary`（灰），`@接收方` 字色 **`--v5-color-success`（#00c261 绿）**——接收方高亮用的是绿色，**不是品牌蓝**（旧口径已废弃）。中间和结尾的连接词（"拍了拍"/"说，……"）14px `--v5-color-text-secondary`，不带容器。
  - 整行不加外层胶囊容器，直接作为一行排在消息末尾（chip 自带底色即可）。
  - **可以链式发生**：场景 Agent 完成自己那部分后，若要再派发给下一个 Agent，用同样的模式在**自己消息末尾**再接一条拍了拍（`@场景AgentA 拍了拍 @场景AgentB 说，……`），不需要每次都回到主 Agent。
  - **全站统一**：此模式**替换**所有原来的居中弱 pill 交接条（无论主 Agent→场景 Agent 还是场景 Agent 之间），不再使用旧样式。
- 场景 Agent 首条消息：完整头像+职能化名称+时间（+ 可选的过程状态标签），再进入「生成中」。**生成态不能同时是最终结果**——产物单独在生成后展示。

## 回答完成操作栏 + 追问建议（`.v5-msg-actions` / `.v5-followup`，见 `components/chat-result-card.html`）

- **只在该轮对话的最终产物之后出现一次**：产物（正文/卡片/列表）→ 完成句（如有）→ **操作栏**→（可选）**追问建议**。不是每条消息都挂，中间的过程态消息（生成中/调用能力）不挂。
- **操作栏 `.v5-msg-actions` 固定顺序**（从左到右，Figma 实测规格）：
  1. 完成状态 pill（`action-check.svg` + 文案，如"已完成当前任务"）——高 38 圆角 19、`--v5-color-bg` 灰底、14px `--v5-color-text-tertiary`；文案按当前任务动态生成，不是写死的通用词；
  2. 播放文字 pill（`action-voice-bars.svg` + "播放文字"）——同尺寸但**白底**；
  3. 复制 / 举报反馈 / 点赞 / 点踩——四个 38×38 圆形纯图标按钮（`action-copy/report/like/dislike.svg`，图标 18px；**点踩 = 点赞图标旋转 180°**），白底、hover/激活换 `--v5-color-bg`。
  **两种视觉分层不要混用**（不要把点赞点踩也做成 pill，或把"已完成"做成纯图标）。
- **追问建议 `.v5-followup`**：可选，只在 AI 判断有值得追问的方向时出现。每条 `.v5-followup__item` 是一句**完整问句**（不是关键词/标签），左侧配一个统一的对话气泡图标，整行可点击、浅底圆角、纵向堆叠 2–4 条。点击即发起该追问（复用本文件顶部的对话生命周期规则，作为新一轮用户提问）。
- **整组可关闭**：追问建议列表下方右对齐一个「× 关闭追问建议」文字链接（`.v5-followup__close`），点击后隐藏整组建议（不影响已出现的操作栏）。关闭是对这一轮追问建议的操作，不是清空对话。
- 追问建议**不与**多 Agent 交接的「拍了拍」混用：拍了拍是任务内部的 Agent 间移交，追问建议是任务结束后给用户的下一步引导，两者可以在同一页面不同轮次出现，但不出现在同一条消息里。

## 状态 → UI

| 状态 | UI |
|---|---|
| 已接收 | 消息头里的过程状态标签（⚡ + 状态词），不单独起一行 |
| 思考中 | 同上，无推理细节 |
| 调用能力 | 同上，标明能力/工具名（如 `⚡ 调用技能：作业完成度分析`） |
| 派发中 | 派发方消息末尾的「拍了拍」mention 条（`@派发方 拍了拍 @接收方 说，……`），不是独立居中 pill |
| 生成中 | 消息头状态标签，说明在生成什么；产物就绪后标签去掉/换成完成态措辞 |
| 需确认 | 橙色/浅警告；一个清晰问题 + 一个主操作（这个仍是独立卡片，不是消息头标签） |
| 暂停 | 低强调灰；保留恢复路径 |
| 完成 | **先产物，后完成句** |

## 并行 Agent 与任务执行系统（copy 真源：`examples/chat-flow-execution.html`）

- 只显示相关 Agent，不列无关 Agent。
- 并行 Agent 的中间完成 = 主对话列里一条短状态行（含来源 Agent）。详细步骤放**执行计划抽屉**，不进主阅读列。
- 多 Agent 返回时，主 Agent 必须总结并指出下一步；单个场景 Agent 的局部备注不能收尾整个任务。

## 内容载体决策（AI 回答「怎么摆」）

| 载体 | 何时用 | 规则 |
|---|---|---|
| 纯文本 | 确认、过渡、短解释、状态 | 一个文本块一个对话动作 |
| 长文排版 | 解释/报告/复盘/有章节 | 标题+分节+块+结尾总结（文档式，**不是卡片堆**） |
| 卡片 card | 对象/任务/资源/结果摘要/可选项 | **一卡一对象**；1 个核心说明；1–3 字段；最多 1 个主操作 |
| 状态 pill/卡 | 待定/暂停/生成/工具调用 | 只表过程，不放结果正文 |
| 抽屉/状态页 | 多字段编辑/多步提交/筛选/发布/高风险确认 | 保留对话壳层，复杂工作在侧层（C 抽屉） |
| 总结块 | 复杂回答结尾 | 一句总体 + 有限下一步，不重复正文 |
| 可浏览列表/表格 | 会议/任务/记录等**多条同构行**、条数不固定（如"列表(10)"） | **不是卡片**：卡片是对象容器，一次装不下会强行内部滚动。见下方「列表类结果不能塞进卡片」 |

### 卡片排布要点（直接回答「AI 回答时卡片怎么摆」）

> 单张卡**内部**怎么长（类别标签/左色条块/选项行/操作行/状态 chip 的统一解剖与五种参考形态）见 `07-card-anatomy.md`；本节管卡片**之间**与卡片 vs 其它载体的取舍。
- **卡片是对象容器，不是长文容器**。需要连续阅读多行 → 用长文排版，别 cardify 每段。
- **情绪支持/心理疏导类回答优先用长文排版，不要 cardify**：把"识别到的信号"“建议”这类内容拆成卡片里的列表项，会让本该温和的陪伴口吻显得像诊断报告，制造距离感。这类内容应该写成自然流动的语言（可以分几句话/几个段落，但不加标题栏和列表符号）。**例外**：真正需要一眼找到、生死攸关的安全信息（如危机热线、紧急联系方式）仍然用卡片承载，保证在紧张状态下也能快速定位——只是这类"安全兜底"卡片要和"情绪疏导"文字分开，不要把两者混进同一张卡。
- 同组卡片**风格一致**，包括容器级样式：同一对话流里的结果卡共用 `.v5-rcard` 外壳。**外壳底色只按「四色生命周期系统」取值**（见 `07-card-anatomy.md`：灰=列表/选择/预填/结束、黄=待开始/待确认、绿=创建成功/已发布、红=警告提醒类，均须配类别标签/状态 chip 文字）——**颜色编码的是状态，不是内容情绪**：不允许因为某张卡内容"比较严重"就临时换成 warning 底、其它卡保持白底的"红绿灯"混搭。非状态类的警示程度仍通过**卡内内容**表达：标题里加图标（⚠️）、关键字段加粗、色条块用 `--warning` 变体。`card.html` 里的语义变体 `--info/--success/--warning/--error` 仍保留给弹窗/表单校验/toast 等**非对话结果卡**场景使用。
- 多卡共享一个操作时用**组级 CTA**（而非每卡一个）。
- **卡片不要撑满对话列宽度**：`.v5-msg-ai__body` 是纵向 flex 容器，块级子元素默认拉伸到父容器全宽（约 800px）。结果卡若不显式限制宽度，字段少的卡（比如只有 2-3 行 key-value）会被拉得很宽，`justify-content:space-between` 一放大就在 key 和 value 之间留出一大片空白，很难看。**结果卡固定加 `max-width:600px; align-self:flex-start`**，让卡片宽度贴合内容、不强行撑满；长文本/列表类正文（`.v5-msg-ai__text`）不用这条限制，保持全宽正常换行。
- 长答顺序：**短引入 → 必要对象/任务卡 → 长文正文 → 总结块**。
- 多卡纵向堆叠或横向滚动；表格/图/注释可作为长文内的局部块，不要每段都做成卡。
- 卡内**不放**表单、筛选、多 CTA、长连续段落；要编辑卡信息 → 经对话生成新卡，或转 C 抽屉。
- **列表类结果不能塞进卡片**：像"会议列表(10)"这种条数不固定、每行结构相同（时间+标题+状态+操作）的可浏览内容，本质是一张表/一个列表页，不是"一卡一对象"的结果卡。常见反模式是把整份列表塞进一张固定高度的卡、内部再加一个滚动条——卡片出现自带滚动条本身就是信号：说明这里该用列表页/抽屉，不该用卡片。正确做法二选一：
  1. **摘要卡 + 查看全部**：卡片里只放前 3–5 条 + 总数（如"会议列表（10）"标题已经暗示还有更多），末尾一个「查看全部」CTA 跳转到独立列表页或 C 抽屉承载完整可滚动列表、支持筛选/分页；
  2. **直接不进卡片**：条数明确且少（≤5 条能一屏看完）时，就是普通结果卡，不需要摘要+跳转这一层。
  卡片本身高度跟随内容，**不设固定高度、不出现内部滚动条**。
- **列表项本身也不能是纯文字堆砌**：像赛事/资源/推荐这类每行有名称+多个属性（级别、标签、状态、时间）的列表，不要写成一行纯文本拼接（`名称 · 属性1 · 属性2 · 属性3`）。每行做成独立的列表项行（边框+圆角容器，标题加粗单独一行，属性用 `.v5-tag` 标签横排），复用「推荐项」结构（`.v5-reco`：`__top` 放标题+可选高亮徽章，下面一行 `.v5-tags` 放属性标签）。
  - **列表项容器底色要和同对话流里的结果卡一致**：`.v5-reco` 类容器背景必须用 `var(--v5-color-surface)`（白），不要用 `var(--v5-color-bg)`（页面底色，浅灰）——用错会让列表项看起来像"透明/没有底"，与旁边的 `.v5-rcard` 视觉不统一。
  - **需要强调的徽章（如"学校重点"/"推荐"这类优先级标记）用实心底+白字**，不要照抄"适配度 92%"那种极浅色浅底文字（`.v5-fit`，`--v5-color-success-bg` 只适合弱提示，非强调场景），否则在截图/弱光环境下容易被当成没设置背景色。强调徽章单独定义（如 `.v5-badge-priority`：`background:var(--v5-color-success); color:#fff`），不要复用弱提示徽章的类名，避免互相污染语义。

### 可视化
- 有趋势/结构/比例/路径/对比/分布/多维度时才用图；图前先给一句核心结论；图与解释同列、不超阅读宽。
- 信息超一列/超 5 个要点/需细看 → 对话里只给摘要图，完整内容移到展开页/抽屉/附件。
- 数据驱动答案必须标口径/来源，**不臆造数据**。简单数据用表格或文本，不为装饰画图。

## 输入台 `@` / `+`（见 input-composer.html）
- 左侧上下文动作：`@`(引用对象：人/Agent/文件/知识库/任务/页面上下文) `+`(添加动作：上传/调用工具/插入模板/选能力)。右侧执行动作：语音/通话/发送。**发送最强、最右、空输入时禁用/弱化**。
- `@` 菜单支持搜索，按 Agent/文件/知识/人/任务分组；每候选有图标+名称+一行描述+可选状态/权限标记；选中变成可移除的引用 pill。
- 运行中的 Agent/工具状态显示在**输入台上方**，不只在对话正文里。菜单不得遮挡正在输入的文字。

## 文件上传 UI（核心交互）

> 对话框传文件样式从 Figma 节点 `3695:6968` 提取（见 `file-upload.html`）；对话中样式见 `input-composer.html` 文件 chip（派生）。

**每个文件卡必含**：文件名 · 类型 · 规模标记(大小/页数/时长) · 状态 · 操作(删除/重试/移除/查看)；视觉文件给缩略图，文档给类型图标。**发送后用户消息里仍保留精简文件卡**，不要只剩「已上传附件」。**文件卡是独立于文字气泡的元素**：不要把文件卡塞进文字气泡内部共用同一个边框/背景——那样会让文字和文件粘成一坨，看不出是两个东西。但也不能各自独立成 thread 子项后被主对话列的大间距（`--v5-space-6`）拉得太远，显得不是一组——正确做法是把「文字气泡 + 文件卡」包一层 `align-self: flex-end` 的纵向 flex 容器（组内小间距，如 6px），让它们紧贴堆叠成一个整体，再作为一个 thread 子项参与主列排布。

### 文件上传状态机
| 状态 | 位置 | UI | 操作 |
|---|---|---|---|
| 已选择 | 输入台附件区 | 类型图标/缩略图+名称+大小+待发 | 删除 |
| 上传中 | 同卡原位 | 进度+上传中；发送可禁用 | 取消 |
| 上传完/解析中 | 同卡 | 解析中+轻进度 | 删除/等待 |
| 已解析 | 同卡 | 就绪可引用；发送转可用 | 删除/发送 |
| 失败 | 同卡 | 红/橙警告+原因 | 重试/移除 |
| 已发送 | 用户消息 | 右对齐用户气泡下方精简文件卡（**只显示文件名**，不带大小/页数/解析状态等元信息——那些是上传管理态才需要的） | 复制/查看 |
| 已引用 | Agent 结果 | 来源标记(页/章/时间/片段) | 查看来源 |

- 单文件卡宽跟随消息/输入台，桌面目标 320–420px（对话中横排卡 270×70）；多文件纵向堆叠或横向滚动，各自保留状态。
- 长文件名中部/尾部截断但保留扩展名。失败留在卡内，默认不弹模态打断。

## 清除记忆（高影响操作）
- 必须说明范围：当前对话 / 当前任务 / 文件引用 / Agent 记忆 / 全局记忆。
- 清除记忆默认**不删可见聊天记录**，除非用户显式选择。
