# HTML 原型绘制规范

## 目标

用统一框架页面绘制当前项目的 HTML / React 原型。先保证页面外壳、导航、目录、对话区、输入区、抽屉容器、管理页结构与已有页面一致，再在内容区补充本次需求。

原型的作用是表达产品逻辑、任务承载方式、关键状态和用户可见结果，不是研发备注页，也不是重新设计页面。

进入项目后，先读取当前项目的 `AGENTS.md`、`MEMORY.md`、`README.md`、`DESIGN.md`、PRD、已有原型目录和页面目录。不得假设项目属于教师中心、教育、医疗或任何固定业务域。所有业务角色、页面名称、主入口名称、PRD 绑定和设计语言，以当前项目文件和用户最新说明为准。

## 核心原则

- **框架页面优先**：创建或修改任何页面前，先寻找同类框架页面或外壳组件。能复用 `Shell`、`Frame`、页面模板或已有同类页面时，不重新手写整页结构。
- **统一画布尺寸**：同一原型工程内所有业务原型页面必须遵循同一画布尺寸和自适应规则。默认以 `1440px` 设计宽度、`750px` 基准画布高度绘制；页面内容不能随意撑大整张画布，长内容应在框架内部滚动或拆分页面。
- **点击跳转拆成页面**：凡是用户可点击进入的配置项、菜单项、详情页、管理页或二级功能页，都必须按独立原型页面注册和呈现；不要只在同一个页面里用组件状态切换伪装成子页面。
- **内容区局部修改**：大部分页面结构应由框架承接，本次需求只改主内容区、卡片区、抽屉内容区或局部状态。
- **同类页面一致**：同一业务域的导航、边栏、顶部区、输入框、对话列、卡片宽度、抽屉宽度、页面背景和间距不因新页面而变化。
- **PRD 口径优先**：当现有原型页面与最新 PRD 冲突时，以最新 PRD 的承载方式、状态定义和完成回流规则为准，不为了保留旧页面而继续沿用过时结构。
- **已确认原型回写 PRD**：如果原型已经根据用户批注或当轮确认被修改，并形成了新的最终展示口径，那么该原型视为最新确认结果；此时必须使用 `prd-writing-style` 同步矫正 PRD，使文档与最终原型一致，不保留相互冲突的双口径。
- **技术可达边界优先**：对话流卡片主体只作为只读展示容器，不在卡片内承载编辑、选择、筛选、刷新等交互；需要修改卡片信息时，通过对话生成新卡片。PRD 明确要求用户最终确认时，可以在只读展示卡片上保留一个主确认按钮。
- **最小改动**：补充用户要求的内容，同时保留已有入口、行为、状态和产品心智。
- **已评审页面变更可追踪**：如果要修改的页面已经在页面目录中标记为绿色，且当前页面的可见原型画面、交互、状态或文案确实发生变化，必须写本次变更记录，在网页中让评审者 / 研发可见，并将该页面状态从绿色改为浅红色，表示“已评审，但是有修改”。
- **最终用户可见**：页面上不能出现开发者备注、实现标签、逻辑标注和原型说明。
- **真实业务可见**：对话区和业务内容区不能直接使用 PRD 模块名、页面编号、路由标题或功能评审标题作为用户可见文案。目录里可以保留评审用标题，但真实业务页面和对话区不应出现这类文档标题；应改成用户真实会说、系统真实会回复的业务话术。
- **不清楚先确认**：如果用户需求比较模糊，或实现方式会违反本 Skill 的框架复用、最小改动、保留旧功能、用户可见文案等要求，必须先向用户确认，不要直接动手。

## 工作流程

1. 阅读当前项目的 `AGENTS.md`、`MEMORY.md`、PRD、用户最新口径、已有页面和组件。
2. 先判断现有页面是否仍然适配最新 PRD；如果页面承载方式、关键状态或完成回流已经落后于 PRD，先改页面结构，再谈局部视觉修补。
3. 如果页面在本轮经过用户批注确认而与 PRD 形成新差异，视该原型为最新确认口径，并使用 `prd-writing-style` 同步回写 PRD；不要只改页面不改文档。
4. 判断需求是否清楚、是否符合本 Skill 要求；如果存在模糊点、冲突点或可能破坏既有框架和旧功能，先向用户确认。
5. 判断页面承载类型：综合入口、对话 / A2UI 卡片、抽屉流程、普通管理页、状态机制页、异常汇总页、素材卡片页。涉及状态或异常时，先判断属于阻断型异常还是模糊预填型异常。
6. 查找同类框架页面：
   - 对话 / A2UI 卡片优先查找当前项目已有的对话壳层、`ChatShell`、`ConversationShell`、`A2UIShell` 或同类页面模板。
   - 综合入口页优先查找当前项目已有的主入口壳层、标签栏入口框架或同类入口模板。
   - Agent 创建 / 管理优先使用既有 Agent 管理页框架。
   - 素材卡片页优先查找当前项目已有的素材页壳层、`CardContainer` 或同类素材页框架。
   - 状态机制页、异常汇总页优先沿用对应主页面的对话壳层和场景标记方式。
7. 如果没有可复用框架，先抽出或创建可复用框架，再填充当前页面内容。不要为单页写一次性整页外壳。
8. 检查页面是否沿用当前工程统一画布尺寸和自适应方式；默认业务页面按 `1440px` 设计宽度、`750px` 基准高度处理，超出内容放入内部滚动区，不让整张画布被内容撑高。
9. 列出本次修改后必须保留的已有元素，例如左侧标签栏、页面目录、搜索、输入框、筛选器、刷新、上传、列表新增、状态提示、附件操作。
10. 修改前检查目标页面在页面目录中的状态；如果页面为绿色，先判断评审者 / 研发在该页面上是否能看到本轮原型变化。只有当前页面自身的画面、交互、状态或文案发生变化时，才记录“这是已评审页面的再次变更”，补充网页可见的变更记录，并在完成后将页面状态标为浅红色。
11. 将新增字段、状态或交互插入最接近的既有区域。
12. 修改后使用 Chrome 插件只读检查或独立无头 Chrome 截图检查页面，确认无白屏、错位、遮挡、文字溢出、旧入口丢失。
13. 如果本次修改影响了承载方式、卡片类型、完成回流、状态机制、字段语义、页面目录或 PRD 绑定，必须使用 `prd-writing-style` 同步调整对应 PRD。
14. 最终回复说明新增内容、保留内容、页面状态变化、变更记录位置、PRD 同步情况和验证方式。

## Chrome 无焦点检查规则

- 原型修改后的页面检查，默认使用 Chrome 插件的只读能力或独立无头 Chrome：打开目标 URL、设置视口、等待加载、截图、读取 DOM / computed style。
- 除非任务必须验证真实点击、输入、登录态、扩展态或当前用户 Chrome 会话，否则不要控制用户正在使用的 Chrome 标签页进行点击、输入、切换标签或激活窗口。
- 如果只需要判断布局、遮挡、文字溢出、白屏、截图差异或滚动内容，优先使用无头 Chrome，避免抢夺用户焦点。
- 如果必须使用用户当前 Chrome 会话，优先复用已有标签页并减少可见操作；涉及点击、输入、切换标签、滚动到用户当前页面等可能抢焦点的动作，应在执行前说明原因。
- 如果 Chrome 插件不可用，可退回到本地无头 Chrome、Playwright 或项目已有检查方式；最终回复必须说明实际使用的检查方式和未完成的验证项。

## 框架页面规则

- 新页面必须先选择框架，再写内容。
- 不得因为一个新功能复制一整套不同导航、不同背景、不同页面宽度或不同输入区。
- 不得在同一原型工程里同时存在多套表现不同的项目主导航，除非 PRD 明确要求不同入口。
- 框架页面里的固定区域默认不改：左侧导航、页面目录、用户头像、主输入区、全局背景、外层滚动容器。
- 只允许在框架暴露的内容区内新增业务卡片、表单、列表、抽屉内容和结果状态。
- 弹窗型页面必须保留原背景页面不变，在整张原型页面上方覆盖一层黑色半透明蒙版，蒙版覆盖范围必须包含当前页面的导航、侧栏、内容区和所有固定区域；弹窗内容以整张原型页面为基准水平和垂直居中展示。弹窗右上角必须提供清晰可见的关闭按钮，使用统一图标或明确文本，点击后返回触发弹窗的父页面或关闭弹窗；关闭按钮不得依赖 hover 才出现，触达区域不小于 40px × 40px，且不能与主操作按钮混淆。不得只覆盖局部内容区，也不得为了表现弹窗而替换、重排、模糊、缩放或改色背景页面。
- 页面跳转需求优先复用现有 `page` 路由参数和目录注册，不额外创造一套跳转机制；跳转入口默认做成固定可见按钮、链接或整行点击，不做依赖 hover 才出现的隐藏交互。
- 点击后会进入另一项配置、另一张详情、另一类资产或另一段独立管理内容时，必须创建独立 `page` 路由和目录项；原页面只保留入口，不在本页内通过本地状态承载完整子页面内容。
- 设置页、管理页、资产页的左侧菜单或顶部标签如果代表可评审的二级页面，菜单项必须链接到对应页面，且每个页面都能独立打开、独立截图、独立同步到 Figma。
- 如果确实需要改框架，需要说明影响范围，并同步检查所有使用该框架的页面。
- 页面命名、目录注册和路由参数应沿用工程已有规则，不另起一套命名体系。

## 页面尺寸与自适应规则

- 同一原型工程内，所有业务原型页必须使用统一的页面尺寸规则；不得因为新增管理页、设置页、长表单页或资源页而单独使用另一套画布宽度、高度或缩放逻辑。
- 默认 HTML 原型画布使用 `1440px` 设计宽度、`750px` 基准高度。若当前项目 `DESIGN.md` 或既有原型工程已经定义了其他明确尺寸，以项目内统一尺寸为准；但同一工程内必须保持一致。
- 外层预览容器负责根据浏览器视口等比例缩放整张 `1440px` 画布；页面内部不要自己按浏览器宽度改变基础设计宽度，也不要让不同页面产生不同缩放基准。
- 原型页面的根画布应固定或等效约束为同一基准高度。常规首页、对话页、管理页、设置页、素材页都应保持相同外层画布尺寸。
- 长对话、长列表、长表单、设置页和管理页如果内容超过基准高度，必须使用内容区内部滚动、左 / 右栏内部滚动、抽屉内部滚动或拆分子页面承载；不要让 `.figma-page`、主画布或外层 Frame 被内容撑高。
- 只有确实表达完整长页面交付、长图导出或 Figma 复刻长画板时，才允许页面高度超过基准高度；这种情况必须是页面目标的一部分，并且需要明确保证底部输入框、页脚或浮层不遮挡最后内容。
- 固定输入框、固定页脚、顶部工具条和浮层应在统一画布内定位；当内容区内部滚动时，这些固定区域不得跟随内容撑高整张画布。
- 横向自适应只发生在外层预览缩放或框架内容区内部布局中；不得出现横向溢出、截图拉伸、文本压缩到不可读或因为容器变窄导致核心字段截断。
- 修改或新增页面时，必须与已有同类页面对比实际渲染尺寸：页面宽度、画布高度、缩放比例、输入区位置和主要内容区起点应保持一致。

## PRD 对照层规则

- PRD 信息只能作为原型评审辅助层出现，不属于业务页面本体；不得把 PRD 说明、全局规则、研发备注或评审文字写进最终用户可见的业务内容区。
- 每个页面只展示与当前页面对应的功能模块 PRD 信息；默认应展示 PRD 中该功能模块的完整原文，而不是只写摘要；不把整份 PRD、全局交互模型、名词表、路由矩阵等全局内容放入 HTML 页面。
- 页面级 PRD 信息应通过外壳或页面目录提供显式开关切换查看；默认不影响页面比例、滚动、输入区、抽屉、卡片和业务内容。
- 状态机制页、异常汇总页等子页面可以继承主页面的功能模块 PRD 信息；如果子页面有独立 PRD 模块，才单独绑定。
- PRD 对照层文案可以面向评审者，但必须与业务页面隔离，避免被误认为用户最终可见文案。
- 新增或调整页面目录时，同步维护页面标题、PRD 编号和页面级 PRD 模块映射；没有明确 PRD 模块的页面不要强行绑定全局说明。
- PRD 原文应尽量从 PRD 文件按章节读取或生成，避免在页面代码中手写一份长期漂移的摘要版本。

## 页面状态与变更记录规则

- 页面目录状态含义固定为：无色表示没做完，黄色表示做完了没评审，绿色表示已评审，浅红色表示已评审但是有修改。
- 修改页面前，必须检查目标页面当前状态。如果页面已是绿色，且当前页面的可见原型画面、交互、状态或文案在本轮确实发生变化，修改完成后必须将该页面标为浅红色，不继续保持绿色。
- 如果本轮只修改了同一功能下的子页面、共享规则、PRD、评审辅助层或不可见实现，而当前主页面从评审者 / 研发视角看没有任何可见变化，则当前主页面不标浅红色，也不显示该变更记录；只把真正发生画面或交互变化的页面标为浅红色。
- 已评审页面被可见修改时，必须新增网页可见的变更记录，方便评审者 / 研发看到当前页面需要重新确认什么内容。
- 变更记录应放在原型评审辅助层中；如果当前项目已有 `PRD 信息`、`变更记录`、页面状态按钮或同类入口，必须沿用，不另起一套评审辅助层。变更记录不得混入最终用户真实业务页面正文、对话气泡、业务卡片或右侧业务抽屉内容。
- 如果当前原型工程还没有变更记录承载组件，应优先补一个轻量的共享承载方式，再写入本次变更；不要只在最终回复或代码提交说明里写变更。
- 变更记录的主文案必须写“当前页面要变更什么内容”，不要写页面分组、功能分类、评审辅助层名称或笼统模块名，避免评审者 / 研发误以为当前页面本体发生了其他变化。
- 变更记录至少包含：变更日期、当前页面的变更内容摘要、是否已完成确认；如项目 UI 约定只保留时间字段，则标题必须承担清晰说明变更内容的职责。
- 变更记录的完成状态必须按“当前页面 + 变更记录”维度保存，并与页面状态联动：当前页面所有变更记录都被标记完成后，页面目录状态应自动恢复为绿色；如果任一已完成记录被还原为未完成，页面目录状态应回到浅红色。
- 已完成的变更记录必须允许还原为未完成，方便评审后发现仍需修改时重新标记；完成和还原都需要同步共享状态，保证其他访问者看到一致结果。
- 同一轮修改影响多个绿色页面时，只给可见原型发生变化的页面分别记录，并分别标为浅红色；未发生可见变化的主页面保持原状态。
- 如果页面不是绿色，可以按实际情况保留原状态；但本轮对页面目录、PRD 绑定或评审辅助层的改动仍应说明。
- 最终回复必须列出被标为浅红色的页面和变更记录展示入口。

## 外壳与 Frame 选择规则

- 先判断页面是不是对话任务，再判断是不是抽屉任务；不要先看页面名字就直接套壳。
- 同一业务域里，主页面、状态机制页、异常汇总页默认跟随同一套承载方式；不要主页面用一种壳，状态页和异常页又换成另一种壳。
- 对话页外壳优先复用已有共享壳层组件，不在页面内部重复手写左导航、会话列表、顶栏和输入区。

### 对话流壳

适用于不打开右侧抽屉、主要在对话区完成理解、追问、展示卡片、状态演示或异常说明的页面。

- 在当前项目已有原型工程中，这类页面优先使用既有对话流壳或同类页面模板。
- 对应共享组件应表现为：左侧主导航 + 会话列表 + 对话顶栏 + 主对话区 + 底部输入区。
- 主页面属于非抽屉流程时，用对话流壳。
- 状态机制页如果表达的是非抽屉流程状态，也用对话流壳。
- 异常汇总页如果表达的是非抽屉流程阻断、追问、空态或失败，也用对话流壳。
- 对话卡片页、查询卡片页、结果卡片页都归到对话流壳；对话流卡片主体不绘制卡内表单、筛选、刷新或局部提交控件。PRD 明确要求最终确认时，只允许一个主确认按钮。

### 抽屉流程壳

适用于任务需要打开右侧已有业务页面或 A2UI 创建页面，在右侧完成校对、补充、选择、发布或继续多步操作的页面。

- 这类页面使用抽屉流程壳，对应共享组件应表现为：左侧主导航 + 对话区 + 原输入区 + 右侧抽屉容器。
- 只有页面真实存在右侧抽屉业务内容时，才使用抽屉流程壳；不能因为某个任务未来可能扩展成抽屉流程，就提前套抽屉壳。
- 抽屉流程主页面用抽屉流程壳。
- 抽屉流程状态机制页、抽屉流程异常汇总页，如果页面本身仍然需要同时展示右侧抽屉，也继续用抽屉流程壳。
- 如果某个状态页或异常页只是解释抽屉流程里的对话区变化，但当前页并不展示右侧抽屉，则回退到对话流壳。
- 如果同一功能的状态机制或异常汇总同时包含阻断型对话场景和需打开抽屉校对的场景，不要混在一个页面里；需要拆成两个页面分别展示，阻断型场景使用对话流壳，抽屉校对场景使用抽屉流程壳。

### 其他 Frame

- 综合入口页使用综合入口框架，不套对话流壳或抽屉壳。
- 普通管理页使用管理页框架，不套对话流壳或抽屉壳。
- 素材卡片页使用素材页框架，不套完整教师端主框架。

## 异常分流规则

绘制状态机制页或异常汇总页时，先按异常能否继续推进来分流。

- 阻断型异常使用对话流壳：缺少课程、缺少班级、无可用候选、无权限、授权未完成、外部连接不可用、前置资源不存在等无法继续推进的场景，都在对话区用简短 AI 回复和少量操作承接，不打开抽屉，不展示抽屉任务卡片。
- 模糊预填型异常使用抽屉流程壳：目标对象、业务范围、资源对象等信息不唯一，但存在可用候选时，不阻断流程；AI 先按最可能对象预填，打开右侧抽屉让用户校对。
- 模糊预填型异常的对话文案要简短说明默认选择，例如 `查到两门物理课，我先按最近上课的《大学物理》填好了，您在右侧确认一下。` 不写长篇推荐依据。
- 模糊预填型异常仍使用空白抽屉任务卡片；卡片不展示候选列表、推荐理由、步骤、进度或过程状态。
- 同一功能同时存在阻断型异常和模糊预填型异常时，必须拆成不同页面或不同目录项表达，例如 `阻断异常汇总` 与 `抽屉异常汇总`；不要把两类场景混在一个异常汇总页里。

## 承载类型规则

### 综合入口页

适用于主智能入口、主 Agent、平台首页或同类综合入口。

- 使用平台已有标签栏或入口框架，不在页面内部重复绘制顶部平台导航。
- 首屏重点展示身份、输入框和智能推荐。
- 智能推荐只在首屏展示；用户发起第一轮对话或点击推荐后，推荐区域收起。
- 推荐文案必须基于具体对象和时间，例如 `要不要看看上周五布置的函数极限作业？`，不写泛泛的 `你可能想查看近期内容`。
- 不加入未确认的会话列表、Agent 切换、长任务推荐或额外能力入口。

### 对话 / A2UI 卡片页

适用于用户通过对话发起任务，系统返回展示卡片、待确认卡片、结果卡片或完成状态卡片。

- 使用统一对话壳层，保留左侧导航、对话列表、顶栏、输入区等固定框架。
- 对话流表达用户发起、AI 理解、卡片出现、用户通过对话确认或修改、结果返回。
- AI 气泡只在 AI 提供新增信息时出现：理解任务、解释推荐、追问、说明阻断原因、给出建议或结论。
- 同一轮 AI 输出只能采用一种主承载形态：单对话、单卡片，或“单卡片 + 单对话”。不要做成“一个 AI 对话 + 多个卡片 / 多个灰底信息块 / 多种模块样式”的堆叠形态；如果是纯文本回答，应在一个对话气泡内部完成轻量排版。
- 纯 UI 状态变化不额外增加 AI 气泡，例如卡片出现、旧卡删除、新卡出现。
- 对话流卡片主体必须是只读信息：不提供卡内编辑、选择、下拉、表单输入、筛选、刷新、展开收起、二次筛选、局部提交或卡内跳转操作。
- 待确认展示卡片在 PRD 明确需要用户最终确认时，可以保留一个主确认按钮，例如 `确认创建`、`确认修改`、`确认取消`。该按钮只表达最终确认，不承载卡内编辑、字段提交或二次选择。
- 用户需要修改卡片信息时，通过对话说明修改内容，系统生成一张新卡片；不要在原卡片内做前端修改。
- 创建或修改类待确认展示卡片出现时，AI 气泡应先说明用户可以通过对话修改，例如 `我先给你看看要创建的信息，你可以通过对话进行修改。` 或 `我先给你看看要修改的信息，你可以通过对话进行修改。`
- 待确认卡片完成后，可以由系统删除旧待确认卡片，再发送一张新的完成状态卡片，避免对话里保留冗余卡片。
- 完成状态卡片必须绑定到指定业务页面或详情入口，展示最终结果和少量固定字段；不要把过程说明、完整变更明细或可编辑内容继续塞进完成卡片。
- 对话流中的结果卡片默认是一次性展示卡，重点是把本轮查询、确认或计算结果直接给用户看清楚。
- 对话流卡片不承载二次筛选、卡内再次查询、条件改写后重新请求或反复刷新；如果用户需要换条件重查，应通过新的用户输入、AI 追问或进入完整页面承接，而不是把卡片做成小型查询台。
- 卡片不承载批量导入、大文件上传、长列表反复筛选、跨页面校对、题目 `换一批` 等复杂后端刷新。

### 抽屉流程页

适用于创建业务对象、导入数据、发布内容、提交审批等复杂流程。

- 对话区先用简短自然语言说明已按用户输入、历史偏好和可用数据做了预填，需要用户在页面中校对；不要展开详细推荐依据。
- 抽屉型任务优先复用当前线上 AI 对话页外壳。外壳只保留左侧导航、顶部区、对话滚动区、原输入框和右侧抽屉容器，不内置固定对话内容，也不内置右侧业务内容。
- 右侧半屏抽屉承载已有业务页面或 A2UI 创建页面，不用对话流卡片假装复杂页面。
- 抽屉右侧可以显示已有业务页面，也可以显示为该任务生成的 A2UI 创建页面；如果需求没有指定抽屉内容区样式，只画抽屉内容框占位即可，不自行设计或高保真绘制业务页面。
- 如果用户提供了真实业务页面地址或截图，优先把现有页面截图放入抽屉内容框作为参考，不要求将真实页面重新 HTML 绘制。
- 如果用户已经将截图保存到本地资产目录，直接引用该资产文件；不要重新截图、裁切、覆盖或另存，除非用户明确要求。
- 抽屉内容区使用页面截图时，截图必须按照抽屉内容区的可用宽度等比例缩放；保持原始宽高比，不得横向溢出、拉伸变形或裁切关键内容。截图高度超过内容区时，使用内容区纵向滚动承载。
- 抽屉内容区页面需要按内容区宽度自适应展示，避免横向溢出或缩放变形。
- 抽屉内容区如果使用 A2UI 创建页面，也需要按抽屉内容区宽度自适应展示，并保留完整校对、补充、提交或发布流程。
- 抽屉流程存在多个步骤或需要多次下一步时，对话区展示空白样式的抽屉任务卡片；卡片不展示业务内容、步骤、进度、推荐理由或说明文案。
- 抽屉流程和异步任务在主对话区必须覆盖两个卡片状态：任务进行时展示过程卡片，任务完成后展示完成卡片。过程卡片只承载任务标题和恢复入口；完成卡片只承载最终结果和少量固定字段。不能只有自然语言提示，也不能只画右侧抽屉而缺少对话区卡片。
- 抽屉任务卡片默认只承载任务标题，例如 `创建作业`、`创建课程`、`导入学生`；不要使用 `右侧任务已打开`、`查看右侧`、`处理中` 这类过程型元文案作为默认标题。
- 抽屉任务卡片在任务未完成时可点击，用于打开或恢复右侧抽屉查看进度；任务完成后置为不可点击。
- 抽屉任务卡片未完成时不展示状态标签，不写 `待校对`、`待补充`、`处理中`、`待处理` 等过程状态；只有任务完成后才在卡片右上角展示一个 `已完成` 标签。
- 抽屉任务卡片一旦出现，后续只允许补充右上角 `已完成` 标签；不要在发布中、校验中、失败重试时把同一张标题卡改写成另一套大卡片结构。
- 右侧业务页面如果已有步骤条、锚点导航或进度引导，按原业务页面截图或占位表达；不要为了抽屉任务卡片另行叠加一套进度条。
- AI 可以预填用户明确给出的信息并推荐选项；无阻断或异常时，不需要在每个步骤持续讲解。
- 涉及发布、导入、删除、取消、开放权限、影响他人或关键业务数据的动作，必须由用户确认。
- 抽屉页面如果需要由前端结束本次任务并触发对话区完成回流，需要在整个抽屉区顶部提供独立且置顶的全宽导航条：左侧使用向右箭头图标表示收起抽屉，右侧提供明确的 `完成` 按钮；点击后由前端关闭或收起本次任务，并在对话区推送完成卡片。完成按钮不要放在抽屉内容卡片标题区内，也不要做成居中的小工具条。
- 抽屉任务完成后，如果 PRD 或用户批注明确要求结构化回流，需要返回完成卡片；不要只用自然语言替代完成卡片。静态主页面需要表达完整生命周期时，可以同时呈现过程卡片的完成态和完成卡片，但过程卡片仍只能保留任务标题与 `已完成` 标签，不增加步骤、进度或解释。
- 中途关闭且业务未完成时，保留抽屉任务卡片可点击状态，方便用户继续打开查看进度。

### 普通管理页

适用于 Agent、Skill、MCP、知识库、工作流、资源、权限等管理页面。

- 使用已有管理页框架，不套用对话卡片结构。
- 重点表达对象关系、管理动作、权限边界和影响范围。
- 创建、编辑、选择、关联、上传、导入、授权、启停、删除等动作需要有对应状态。
- 资产级操作和关联级操作要分清：资产级影响全局资产池，关联级只影响当前对象。
- 标准库、定制库、自建内容等来源差异需要在界面状态中体现。

### 素材卡片页

适用于单独展示可复用卡片资产。

- 使用素材页框架，不使用完整对话框架。
- 只展示卡片本体及必要标题，不额外绘制平台导航。
- 卡片宽度、圆角、边框、阴影、标题层级和状态标签沿用同类素材卡。

### 状态机制页

当 PRD 存在正式状态机制，且状态不少于 3 行或存在分支状态时，需要绘制独立状态机制页。

- 使用对应主页面的框架。
- 每个 PRD 状态行至少对应一个场景。
- 场景顺序与 PRD 表格从上到下严格一致。
- 场景标签可包含状态编号，但状态编号不得出现在用户气泡和业务卡片正文中。
- 手动触发或系统回调状态不虚构用户气泡。

### 异常汇总页

当一个功能存在多个空态或异常场景时，使用一个异常汇总页集中展示。

- 使用对应主页面的框架。
- 如果异常场景需要不同外壳，必须拆分为多个异常汇总页；例如阻断型异常使用对话流壳，模糊信息但可默认预填的异常使用抽屉流程壳。
- 阻断型异常只展示对话阻断链路；模糊预填型异常展示 AI 简短说明、空白抽屉任务卡片和右侧抽屉。
- 每个场景展示完整链路：时间戳、用户气泡、必要的 AI 气泡、结果卡片或阻断提示。
- 场景标签写业务语义，不写开发者说明。
- PRD 明确不是空态的分支，不命名为某某空态。

## 卡片规则

- 子页面中的编辑型卡片必须从主页面主卡片或共享组件出发，不重新手写相似卡片。
- 同一业务卡片保持同一套容器、标题区、筛选区、列表行、按钮层级、圆角、阴影、间距和色彩。
- 抽屉任务卡片属于共享组件，默认态、进行中、异常态都从同一组件分支演化；不要在各页面单独拼一套视觉结构。
- 对话流里的查询结果卡、详情结果卡、结果卡和完成状态卡片全部信息化展示，只保留本轮结果展示所必需的信息，不保留二次筛选器、条件输入框、提交按钮或“再次查询”按钮。
- 待确认展示卡片的信息区保持只读；PRD 明确要求最终确认时，允许一个主确认按钮，例如 `确认创建`、`确认修改`、`确认取消`。不要把这个按钮扩展成表单提交、字段保存、二次筛选或局部编辑入口。
- 创建 / 修改类任务如果 PRD 定义为展示卡片，主页面不得画成表单、管理卡或抽屉流程；使用预填摘要卡或变更摘要卡 + 单一确认按钮 + 完成状态卡片承接。
- 创建类摘要卡展示标题、时间、对象范围、必要备注等确认前信息；修改类摘要卡只展示本次将要修改的字段及修改前后对比，不展开未变化字段。
- 对话流卡片的信息修改只能通过对话触发新卡片，不在原卡片中做字段编辑或局部刷新。
- 对话流卡片支持生命周期替换：待确认卡片完成后，可以移除旧卡片并展示新的完成状态卡片。
- 完成状态卡片出现前，需要删除、隐藏或视觉上收起之前的操作卡片，避免用户同时看到待确认卡片和完成卡片两套状态。
- 完成状态卡片可绑定详情页或业务页面，但不要在卡片底部额外展示“点击卡片查看详情”“进入详情”这类说明行；卡片整体可点击或使用必要入口即可，避免所有完成卡片重复出现说明文案。
- 会议、日程等创建和修改主页面如果属于对话流展示卡片，应显示待确认展示卡片和确认按钮；确认后返回完成状态卡片，不使用完整交互表单承载。
- 特定业务对象的展示限制以当前项目 PRD 为准；例如项目要求隐藏地点、人员、资源名称等敏感信息时，不得在卡片或页面中展示。
- 新增字段沿用已有行标签、输入区、选择器、上传行、开关、折叠区样式。
- 条目级字段放在对应列表行内，不移动到全局表单区。
- 完成态、结果态、只读摘要态使用展示型结果卡片，不继续展示可编辑主卡片。
- 编辑型卡片只适用于非对话流页面或抽屉内容区；对话流内不绘制编辑型卡片。歧义、推荐依据、默认策略由 AI 气泡说明，卡片内只用必要的展示文案表达。
- 多候选或目标锁不定时，优先在原业务卡片顶部用选择器或候选列表处理，不拆出独立空白选择卡。
- 缺少必填前置数据且无法推进时，使用简短 AI 问题和少量操作，不渲染完整禁用业务卡片。
- 如果任务只需要用户最终确认，优先使用待确认展示卡片；如果需要用户补充、编辑、选择、上传或多步提交，应进入抽屉流程或 A2UI 创建页面。

## 交互状态规则

- 所有可交互按钮和控件都要有结果状态：打开面板、选择完成、展开收起、提交中、提交成功、提交失败、字段校验。
- 对话流卡片不适用卡内交互状态；只需要表达卡片出现、旧卡删除、新卡出现、完成状态卡片返回等生命周期变化。
- 不把多个卡片内交互状态堆在主页面；主页面保持主流程，复杂交互拆到子页面。
- 开关或折叠会影响业务流程时，需要展示收起态和展开态。
- 依赖查询数据、可选记录或前置字段的页面，必须绘制空态或兜底态。
- 缺少目标对象、权限对象、业务范围这类前置条件时，优先使用简短 AI 引导 + 少量选择按钮或操作按钮，不额外弹出说明型任务卡片。
- 有可恢复默认数据或候选数据时，优先在原业务卡片中展示，不停留在纯 AI 气泡。
- 对话流卡片不表达卡内校验错误；需要校对或补充时由 AI 气泡追问，或进入抽屉 / A2UI 创建页面处理。
- 页面内前端校验即可拦截的问题，不额外升级成对话区异常场景或失败卡片；异常汇总只保留真实阻断主流程的情况。

## 文案与视觉规则

- 原型文案默认中文；产品名、协议名、代码标识、文件名、组件名和已确认专有名词可以保留原文。
- 页面文案必须面向最终用户，不出现 `交互态`、`次级页面`、`入口状态`、`ActionMarker`、`用户点击` 等原型说明。
- 页面文案必须写成用户在界面里真实会看到的话；不要写“右侧保留某页面”“这里展示某流程”“页面内每个关键操作都有结果”“当前页只做占位”这类面向设计评审或研发说明的句子。
- 对话区尤其不能把 PRD 章节名、页面目录名、功能编号或内部承载方式写成可见标题。如果需要表达任务，使用当前项目中的真实业务名称，例如 `开设课程`、`创建课程`、`导入学生` 这类用户能理解的任务名称。
- 即使是 P2 占位能力，页面文案也应写成面向用户的可见状态，例如“该功能暂未开放”或“暂时还不能直接处理这类任务”，不要把“后续版本补充”“能力占位”“按最新规则补充”直接写进页面正文。
- 等待文案要自然、简短，不写“正在识别任务承载方式”这类产品内部判断过程，也不把系统查询步骤拆得过细；优先使用“我先整理一下信息”“正在确认发布信息”这类用户能理解的话。
- 用户提供的截图优先作为字段、层级和业务语义参考，默认不视觉复刻。
- 如果用户明确给了可复用的卡片参考样式，优先提炼其结构层级和视觉节奏用于同类卡片：例如左侧图标块、右侧标题/时间两行结构；保留当前业务语义，不照搬无关文案。
- 不使用抽象总结卡替代真实 UI 流程。
- 默认任务卡的边框、圆角、阴影和留白要克制，避免为了强调任务而做出过大的外框、过厚边线或过重阴影。
- 不隐藏已有筛选器、刷新、比例条、状态提示、附件操作、列表新增按钮等旧控件。
- 文本不能溢出、遮挡或被压缩到不可读；空间不足时调整布局、字号或文案长度。
- 页面色彩、字体、圆角、阴影、边框和间距优先遵循当前项目既有 UI 风格、`DESIGN.md`、设计系统文档、Figma 抽取规范或已有页面视觉规则；没有明确设计规范时，先从同类页面归纳，不跨项目照搬。
- 如果页面后续需要同步到 Figma，关键图标必须使用可迁移资产表达。Logo、头像、工具栏图标、Agent / Skill 按钮、发送、附件、语音等关键图标，不要只依赖外链 SVG 的 CSS 变量填色、CSS `filter` 改色、`mask`、伪元素、图标字体或背景组合来呈现。应使用内联 SVG，或使用已经写死可见 `fill` / `stroke` 的本地 SVG / PNG；不同状态需要不同颜色时，优先准备显式颜色的资产或内联 SVG 状态样式，而不是用 `filter` 临时改色。

## 权限与数据规则

- 不展示当前用户无权限访问的数据。
- 学生个人信息、成绩、提交记录和学习分析需要最小化展示。
- 授权未完成、连接不可用、数据缺失、提交失败、权限不足都要有可见状态。
- MCP、外部资源、API 调用等敏感配置不明文展示密钥。
- 高影响动作需要明确确认状态，例如发布、导入、删除、取消、开放权限、启用外部连接。

## Figma 复刻衔接规则

- HTML 原型已经被用户确认或浏览器截图验证正确时，Figma 复刻失真属于导入结果问题，不得为了补偿 Figma 导入偏差反向修改 HTML / CSS / React 源码。只有原型本身错误、用户明确要求修改原型，或确实需要临时隔离的捕获入口时，才改原型源码。
- Figma 复刻完成后必须检查渲染截图，不能只看节点存在、图层数量、文字数量或边界数据。节点存在但被遮挡、透明、空白、错位、放到错误父级或没有正确渲染时，仍然视为复刻失败。
- 如果关键图标在浏览器中可见，但同步到 Figma 后变成空 `Image`、白色不可见、缺失或只剩按钮文字，应判定为资产可迁移性问题。先检查是否使用了外链 SVG 变量填色、CSS `filter` 改色、`mask`、伪元素或图标字体；如果 HTML 资产表达本身不可迁移，应先改成显式颜色的 SVG / PNG / 内联 SVG 再复刻。若 HTML 原型已确认不能再改，则只在 Figma 中按原型截图修复资产，不要通过改变尺寸、间距或布局来掩盖图标缺失。
- 复制 Figma 图层、图标、Logo、Mask group 或组件后，必须检查父级布局是否触发重排。重点核对父级 `layoutMode`、复制节点的 `x/y`、右侧工具区位置、层级顺序和可见边界；如果复制后左右位置互换或漂移，直接在 Figma 中修正父级布局和节点位置。
- 长对话流、长列表、长表单和多轮卡片页面复刻到 Figma 时，画板高度应覆盖完整业务内容。底部输入框、固定页脚或浮层应放在完整内容之后，不能遮挡最后一张业务卡片、最后一段回复或最后一个表格行；不要把长内容压缩进首屏高度。

## 回归检查清单

完成前检查：

- 是否优先复用了同类框架页面或壳层组件。
- 固定区域是否保持一致：导航、目录、输入区、外层背景、页面宽度、抽屉位置。
- 弹窗型页面是否保留了原背景页面不变，并在整张原型页面上方覆盖黑色半透明蒙版；弹窗是否以整张原型页面为基准水平和垂直居中，而不是只在内容区、右栏或局部容器内居中；弹窗是否提供常显的关闭按钮或取消入口，且位置清晰、触达区域足够、可以返回父页面。
- 页面是否遵循当前工程统一画布尺寸：默认 `1440px` 设计宽度、`750px` 基准高度，且没有因为内容过长把整张画布撑高。
- 长内容页面是否使用内容区、左右栏、抽屉或列表内部滚动承载，而不是改变页面外层尺寸或导致与其他页面缩放比例不一致。
- 页面在常见浏览器视口下是否通过外层预览等比例自适应，且没有横向溢出、局部拉伸、文字压缩或固定输入区遮挡内容。
- 已有主入口和关键操作是否仍然存在。
- 新字段是否插入在最接近的既有区域，没有挤掉旧功能。
- 页面可见文字是否全部面向最终用户。
- 对话区是否没有出现 PRD 模块名、页面编号、路由标题、承载方式名称或评审标题。
- 对话页是否只在 AI 提供新增信息时出现 AI 气泡。
- 对话流卡片主体是否只读，没有卡内输入、选择、筛选、刷新、局部提交、展开收起或跳转操作。
- 如果待确认展示卡片有按钮，是否只保留了 PRD 明确要求的单一最终确认按钮，例如 `确认创建`、`确认修改`、`确认取消`。
- 如果用户需要修改卡片信息，是否通过对话生成新卡片，而不是在原卡片中编辑。
- 待确认卡片完成后，是否支持删除旧卡并返回新的完成状态卡片，避免冗余信息堆叠。
- 创建 / 修改类主页面如果 PRD 要求展示卡片，是否使用只读摘要卡 + 确认按钮 + 完成状态卡片，而不是表单、管理卡或抽屉流程。
- 特定业务对象是否遵循了当前项目 PRD 中的敏感信息、地点、人员、资源名称等展示限制。
- 卡片页是否复用主卡片或共享组件。
- 抽屉流程是否有打开、右侧页面承载、空白抽屉任务卡片、中途关闭、自然语言完成回流状态。
- 抽屉流程和异步任务的对话区是否同时覆盖过程卡片和完成卡片，而不是只有抽屉、只有提示语或只有完成结果。
- 抽屉内容区是否支持已有业务页面或 A2UI 创建页面两种承载方式，并按内容区宽度自适应展示。
- 多步骤抽屉任务卡片是否不展示业务内容、步骤、进度、推荐理由或说明文案。
- 多步骤抽屉任务卡片默认态是否只保留任务标题，并避免出现过程型元文案。
- 多步骤抽屉任务未完成时卡片是否可点击且无状态标签，任务完成后是否不可点击并仅在右上角展示 `已完成` 标签。
- 如果使用了用户提供的卡片参考样式，是否只复用了结构和视觉层级，没有把参考文案机械带入当前业务。
- 用户已保存截图到本地资产目录时，是否直接引用该资产文件，且没有重新截图、裁切、覆盖或另存。
- 抽屉内容区如果使用页面截图，截图是否按内容区宽度等比例缩放，且没有横向溢出、拉伸变形或关键内容裁切。
- 多步骤抽屉是否保留真实业务页面已有的步骤条、锚点导航或进度引导，且没有额外叠加一套进度条；未指定内容区时是否只使用抽屉内容框占位。
- 普通管理页是否没有误套对话卡片结构。
- 状态机制页和异常汇总页是否与 PRD 表格逐行对应。
- 状态机制页和异常汇总页如果涉及不同外壳，是否已拆成多个页面分别展示。
- 阻断型异常是否使用对话流壳，且没有打开抽屉或展示抽屉任务卡片。
- 模糊预填型异常是否使用抽屉流程壳，AI 是否先预填最可能对象并提醒用户在右侧校对。
- 交互按钮是否有结果状态。
- 空态、异常态、权限不足和提交失败是否覆盖。
- 页面是否无白屏、错位、遮挡和文字溢出。
- 是否使用 Chrome 插件只读检查或无头 Chrome 截图完成页面验证；如果用了其他方式，是否说明原因。
- 是否避免了不必要的点击、输入、切换标签或激活用户当前 Chrome 窗口。
- 如果本轮涉及 Figma 复刻，是否确认 HTML 原型本身正确；若只有 Figma 导入失真，是否直接修复 Figma 画板而没有污染 HTML 原型源码。
- 如果本轮涉及 Figma 复刻，是否查看了 Figma 渲染截图，而不是只用节点元数据、图层数量或文本数量判断完成。
- 如果本轮涉及 Figma 复刻，关键 Logo、头像、按钮图标、工具栏图标是否已避免依赖 CSS `filter`、`mask`、外链 SVG 变量填色或伪元素；是否能作为显式可见的 SVG / PNG / 内联 SVG 被捕获。
- 如果从 Figma 其他画板复制图标、Logo、Mask group 或组件，是否检查并修正了父级布局、节点位置、工具区位置和层级顺序。
- 如果复刻长对话流或长内容页面，是否保留完整内容高度，并确认底部输入框、页脚或浮层没有遮挡最后的业务内容。
- 如果修改了绿色页面，当前页面是否真的发生了评审者 / 研发可见的画面、交互、状态或文案变化；如果没有，是否保持原状态且不显示变更记录。
- 如果绿色页面确实发生可见变化，是否已在网页中新增评审者 / 研发可见的变更记录，并将页面目录状态从绿色改为浅红色。
- 变更记录是否直接说明当前页面要变更什么内容，而不是写页面分组、功能分类、评审辅助层名称或笼统模块名。
- 变更记录标记完成后页面状态是否自动恢复绿色；还原为未完成后是否回到浅红色。
- 变更记录是否只出现在原型评审辅助层或共享变更记录组件中，没有混入最终用户真实业务页面正文。
- 如果本轮修改影响了承载方式、卡片类型、完成回流、状态机制、字段语义、页面目录或 PRD 绑定，是否已使用 `prd-writing-style` 同步调整 PRD。
- 如果项目存在构建命令，页面仍能构建或通过类型检查。

## 最终回复

说明新增或修改了什么，明确保留了哪些框架和旧功能，并说明执行了什么验证。如果修改了绿色页面，需要说明哪些页面因为可见原型变化已标为浅红色、哪些相关页面因无可见变化保持原状态、变更记录在哪里可见。如果本轮同步过 PRD，需要说明同步了哪个 PRD；如果本轮没有同步 PRD，需要说明没有触发 PRD 同步的原因。
