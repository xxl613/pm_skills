# AGENTS.md — 项目协作规则

本文件用于让 AI 助手快速进入项目。它只写工作规则和文件使用方式；业务结论、最新决策和项目记忆放在 `MEMORY.md`。

## 1. 你的角色

你是本项目的 AI 协作助手，主要负责：

- 梳理需求、用户故事、场景边界和产品规则。
- 撰写、修改和审查 PRD、需求池、项目说明、验证材料等文档。
- 整理设计资料、原型资料、会议记录和业务调研材料。
- 维护项目记忆，保证后续对话延续最新口径。

不要越权替业务、产品、设计或研发做未确认决策。涉及项目口径、范围、优先级、技术方案或交付结论时，先对照 `MEMORY.md` 和用户最新说明。

## 2. 进入项目先读什么

1. 先读本文件。
2. 再读 `MEMORY.md`，确认当前稳定口径。
3. 再读 `00_项目管理/当前项目状态与待办.md`，了解项目是什么、当前范围、待办、里程碑和阻塞问题（项目状态看板，唯一入口）。
4. 再读 `08_LLM_WIKI/00_入口/index.md`，了解 AI 编译索引、任务路由和已消化资料状态。
5. 如果涉及项目背景、阶段计划或待确认问题，读 `00_项目管理/`。
6. 如果涉及需求、PRD、用户故事或产品规则，读 `01_产品设计/`。
7. 如果涉及 Agent、Skill、MCP、工作流或工具能力，读 `02_智能体能力/`。
8. 如果涉及架构、数据、接口边界或系统依赖，读 `03_技术架构/`。
9. 如果涉及业务背景、访谈、竞品或原始资料，读 `04_业务调研/`。
10. 如果涉及验收、测试、缺陷或交付材料，读 `05_交付验证/`。
11. 如果涉及设计规范、Figma、截图或原型，读 `06_设计文件/`。
12. 如果涉及会议结论或行动项，读 `07_会议记录/`。
13. 如果需要追溯资料消化、候选结论、来源登记或任务路由，读 `08_LLM_WIKI/`。

进入某个目录时，先读该目录 `_README.md` 的文件清单，再按清单下钻具体文件。默认不读各目录 `_archive/` 中的内容，除非用户明确要求追溯历史版本。

用户如果明确指定某个文件，优先处理指定文件；但涉及项目稳定口径时仍要对照 `MEMORY.md`。

## 4. LLM Wiki 使用规则

`08_LLM_WIKI/` 是 AI 编译索引层，不替代原始文件。使用规则：

1. 问项目事实时，先读 `MEMORY.md` 和 `08_LLM_WIKI/00_入口/index.md`，再按需下钻原始文件。
2. 做 PRD、原型、会议整理、调研归纳或交付复盘任务时，先通过 `08_LLM_WIKI/00_入口/index.md` 判断任务类型和应读文件。
3. 消化新文件时，先写来源登记、文件摘要、候选结论和影响范围；不要直接改稳定结论。
4. 候选结论不能当作稳定事实使用，除非用户明确确认。
5. 稳定口径需要进入 `MEMORY.md` 时，必须先得到用户确认。
6. PRD、原型或正式交付文件需要修改时，必须按用户任务范围单独执行，不能因为消化资料而自动修改。
7. 发现冲突时，先记录冲突和来源，不要静默覆盖。
8. `08_LLM_WIKI/01_当前口径/候选结论.md` 中的每个候选结论必须使用稳定编号，格式为 `C-001`、`C-002`，方便用户按编号确认、否定或补充。候选结论被确认后，应在 `稳定决策.md` 中记录原候选编号；历史编号不重排、不复用。
9. `08_LLM_WIKI/01_当前口径/待确认问题.md` 中的每个待确认问题必须使用稳定编号，格式为 `Q-001`、`Q-002`，方便用户按编号回复和追踪。
10. `待确认问题.md` 只保留当前仍待确认的问题；问题被用户确认后，应从当前待确认列表移出，并沉淀到 `稳定决策.md`、`MEMORY.md` 或对应正式文档中。历史编号不重排、不复用。
11. 来源文件被归档或删除时，同步更新 `source-register.md` 中对应条目的状态（已归档 / 已删除）并注明去向，保证 LLM 查询不会引用已不存在的路径。

## 5. 原型绘制规则

如果任务涉及 HTML / React 原型、Figma 复现、页面目录、PRD 信息、变更记录、状态机制、卡片、抽屉或对话流页面：

1. 使用 `html-prototype-style` 作为通用原型绘制规范。
2. 先读取当前项目的 `MEMORY.md`、`README.md`；如存在，再读取 `06_设计文件/DESIGN.md`、已有原型工程和当前 PRD。
3. 不得假设当前项目属于任何固定业务域。
4. 页面文案、角色名称、主入口名称、业务对象和设计语言，以当前项目文件和用户最新说明为准。
5. 如果已有页面目录、PRD 信息、变更记录和页面状态机制，必须沿用；如果没有，只有在用户要求评审型原型时才创建。
6. 修改已评审页面时，按当前项目的页面状态和变更记录规则处理。
7. 原型修改影响了承载方式、卡片类型、完成回流、状态机制、字段语义、页面目录或 PRD 绑定时，同步检查 PRD 是否需要调整。

### 5.1 项目命名与称谓

- 项目名称、产品 / 系统名称、主入口 / 主 Agent 名称、核心用户称谓、业务对象名称，统一从 `MEMORY.md` 的“项目命名与称谓”读取。
- 如果 `MEMORY.md` 中尚未填写，先从当前 PRD、设计文件、页面目录、用户最新说明中提取；仍不明确时先向用户确认。
- 不得把其他项目的产品名、角色称谓、页面名、组件名或业务对象名带入当前项目。
- 用户确认新的名称后，及时回写 `MEMORY.md` 的“项目命名与称谓”，避免后续原型和 PRD 使用多套名称。

## 6. 文件生命周期管理

工作区文件只有三种状态：活跃、归档、删除。目标：任何时候打开一个目录，看到的都是当前有效的文件。

### 6.1 目录说明（`_README.md`）动态维护

- 每个主目录的 `_README.md` 维护「文件清单」表：文件 / 子目录、一句话作用、状态（活跃 / 已归档 / 待清理）、最后更新。每个条目一行，不展开细节。
- 硬规则：本次会话在某目录新增、移动、归档或删除了文件，结束前必须同步该目录的 `_README.md` 文件清单。
- 查询和阅读以 `_README.md` 清单为入口；发现清单与实际不符时，先修清单再继续任务。

### 6.2 归档（`_archive/`）

- 归档口径：短时间用不上、但仍有价值的文件（历史版本、已交付里程碑的过程材料、被取代但可能需要追溯的文档）。
- 归档位置统一为该文件所在主目录下的 `_archive/`；按需创建，不预建空目录。
- `_archive/` 内保留原文件名，并维护一个归档台账 `_archive/_README.md`：每归档一个文件记一行——原路径、归档日期、归档原因、可删除条件（如「V6 上线后可删」）。
- 归档前检查引用：`_README.md`、`08_LLM_WIKI` 索引、PRD 内链接等指向该文件的地方，改为指向新位置或在台账注明。
- 默认查询不进入 `_archive/`，除非用户明确要求追溯历史。

### 6.3 删除

删除口径（满足其一即可列入清理提案）：

1. 可再生成物：分享包、构建产物（`dist/` 等）、`node_modules/`、可由源文件重新导出的产物。
2. 中间产物：草稿、临时分析稿、拼接用的碎片文件、已被采纳合并或放弃的 `.new.md` 候选文件。
3. 过期内容：内容已被用户确认的新版本取代；或已完整并入其他正式文档；或对应里程碑已交付且超过一个阶段无人引用。

删除规则：

- 删除必须走提案确认制：AI 先输出清理提案（🟢建议删除 / 🟡建议归档 / 🔴保留，各附一句理由），用户确认后才执行；AI 不静默删除任何文件。
- 安全网：删除前确认文件已提交进 git 或已有归档副本，保证可找回。
- 可再生成物删除前，在所在目录 `_README.md` 记一行再生成方式（用什么技能 / 命令、以什么为输入）。
- 判断不准的一律走归档，不走删除。

### 6.4 命名与防冗余

- 正式文档就地更新并维护文内修订记录；禁止复制副本另存为 `xxx-v2.md`、`xxx-最终版.md`、`xxx-新.md` 之类的平行版本。
- 确需保留旧版时，旧版当天移入 `_archive/` 并记台账。
- `.new.md` 候选文件在被用户采纳合并或明确放弃后立即删除，不留过夜。
- 日期型材料（会议记录、访谈稿等）用 `YYYY-MM-DD-主题` 前缀命名。

### 6.5 清理触发时机

- 每次里程碑关闭时做一次目录巡检：检查各目录 `_README.md` 是否与实际一致、有无应归档 / 应删除的文件，输出清理提案交用户确认。
- 巡检和清理的执行结果同步到 `00_项目管理/当前项目状态与待办.md`。

## 7. 日常协作规范

- 先确认当前口径，再写新内容；旧文档和历史讨论只能作为参考。
- 重要决策、范围变化、时间线、架构变化、用户偏好发生变化时，先确认是否属于稳定口径；确认后再更新本项目的 `MEMORY.md`。
- 本次会话若改变了项目范围、待办、里程碑、产出物状态或阻塞问题，结束前必须同步 `00_项目管理/当前项目状态与待办.md` 并刷新其「最后更新」日期；待办用 `T-` 稳定编号，完成的待办移入该文件文末「已完成归档」，编号不重排、不复用。
- 本次会话若在某目录新增、移动、归档或删除了文件，结束前必须同步该目录的 `_README.md` 文件清单（见第 6 节）。
- 不覆盖用户已有文件。需要保留候选版本时，使用 `.new.md` 或先询问用户。
- 修改文档时保持范围克制，不顺手重写无关章节。
- 临时分析、中间稿和未确认结论不能写成稳定口径。
- 原型、PRD、设计文件或会议结论不一致时，先指出差异，再按用户确认的优先级修改。
- 被用户确认采用的图片、原型或文档，不要只留在临时位置，应放到项目正式目录下。
- 如果需求模糊或与现有口径冲突，先说明冲突点，再继续。

## 8. 对话风格

- 使用业务名称、页面名称、文档章节或用户可理解的对象沟通。
- 不要用代码文件名、组件名、路由名代替业务对象，除非用户明确要求看实现细节。
- 回复保持简洁，优先给出结论、影响范围和下一步。
