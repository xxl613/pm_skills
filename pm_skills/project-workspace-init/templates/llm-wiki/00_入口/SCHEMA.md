# LLM Wiki 维护规则

## 1. 定位

`08_LLM_WIKI/` 是项目资料的 AI 编译层，用于记录任务路由、来源登记、文件摘要、候选结论、待确认问题和追溯关系。

它不是原始资料库，也不是稳定事实的唯一来源。

## 2. 默认可信顺序

遇到冲突时，默认按以下顺序判断：

```text
用户最新确认
> MEMORY.md
> AGENTS.md
> 当前主 PRD
> 当前原型状态
> 整理后会议纪要
> temp 临时材料
> 旧 PRD / 旧原型
```

`AGENTS.md` 主要约束协作方式，`MEMORY.md` 主要记录稳定项目事实。业务结论冲突时以 `MEMORY.md` 为准；工作方式冲突时以 `AGENTS.md` 为准。

## 3. 写入规则

- 消化文件时，可以写入来源登记、文件摘要、候选结论、影响范围和待确认问题。
- 候选结论必须使用稳定编号，格式为 `C-001`、`C-002`。
- 待确认问题必须使用稳定编号，格式为 `Q-001`、`Q-002`。
- 候选结论不能直接当作稳定事实使用。
- 稳定结论必须由用户明确确认后才能写入。
- 已确认的待确认问题不继续保留在 `待确认问题.md` 的当前列表中，应迁移到 `稳定决策.md`、`MEMORY.md` 或正式文档。
- 历史编号不重排、不复用。稳定决策需要记录原候选编号或原问题编号。
- `MEMORY.md`、主 PRD、原型和正式交付文件不能因为文件消化而自动修改。
- 发现冲突时，先登记冲突、来源和影响范围，再等待用户确认处理方式。

## 4. 文件消化流程

当用户要求消化文件时：

1. 读取原文件。
2. 判断文件类型和可信层级。
3. 更新 `08_LLM_WIKI/06_来源登记/source-register.md`。
4. 更新 `08_LLM_WIKI/06_来源登记/file-digests.md`。
5. 如有可复用判断，更新 `08_LLM_WIKI/01_当前口径/候选结论.md`。
6. 如有不确定事项，更新 `08_LLM_WIKI/01_当前口径/待确认问题.md`。
7. 如有冲突，记录冲突和替代关系。
8. 只有用户明确指定哪些条目可提升时，才更新 `稳定决策.md`；不得自行提升候选结论。
9. 用户确认某个 `Q-` 问题后，将其从 `待确认问题.md` 移出，并在 `稳定决策.md` 中记录原问题编号。
10. 用户确认某个 `C-` 候选结论后，在 `稳定决策.md` 中记录原候选编号，并按影响范围同步 `MEMORY.md` 或正式文档。

## 5. 禁止事项

- 不自动扫描文件系统。
- 不自动把新文件写成稳定结论。
- 不自动修改 `MEMORY.md`。
- 不自动修改 PRD、原型或会议原文。
- 不复制其他项目的业务内容。
