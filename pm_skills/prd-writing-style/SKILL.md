# 教师中心 PRD 编写范式

## 目标

用用户偏好的方式编写和修改教师中心 PRD：直接、面向需求、便于研发拆解，不写未经确认的研发实现细节，不把原型已经能表达的控件清单重复写成 PRD。

本 Skill 主文件负责通用判断、PRD 修改流程、普通页面结构、路由版综合 PRD 结构和研发拆解规则。卡片设计专项规则不默认加载；只有判断为传统对话 / A2UI 卡片详细能力，或用户明确要求卡片状态、S1、AI 气泡、卡片空态、状态机制时，才读取同目录下的 `card-design-rules.md` 并遵循其中规则。如果 `card-design-rules.md` 与本文件的技术可达边界冲突，以本文件为准。

## 工作流程

1. 识别目标文档类型：PRD 章节、需求池条目、功能说明、研发对齐文档、路由版综合 PRD。
2. 读取目标文档和用户最近确认的产品口径；新口径优先于旧文档。
3. 判断文档对象属于哪一类：
   - 对话 / A2UI 卡片类能力
   - 抽屉流程类能力
   - 普通页面 / 后台管理类能力
   - 路由版综合 PRD
4. 判断优先级：
   - P0 / P1：必须补齐任务承载方式、关键规则、权限边界和验收标准。
   - P2：只保留占位，记录功能名称、承载方向和一句话边界，不展开详细流程、字段、状态和专项验收。
5. 除非用户要求重组结构，否则保留现有文档层级和表格风格。
6. 写清产品行为、边界、输入、输出、状态、触发点、兜底策略、权限影响和验收标准。
7. 删除过时口径、重复样板、作者日志、执行说明和无关解释。
8. 完成前检查标题编号、表格一致性、术语一致性和 P2 是否仍为占位。

## 风格规则

- 使用直接的需求语言。
- 优先使用需求条目或表格，不写长篇解释段落。
- 示例要短，并且贴近真实教师任务。
- 重组功能列表时，按用户可感知的业务场景分组。
- 中文产品语言要自然；已确认的专有技术名词可保留原文。
- 正文默认使用中文；只有产品名、协议名、文件名、组件名、已确认专有名词可以保留原文。
- 不写接口字段、后端枚举、JSON 路径、实现变量名和研发过程。
- 不写视觉还原项；原型能表达的布局、按钮、图标、控件细节不重复写。
- 不把“创建页”“管理页”“配置页”默认写成卡片类能力。
- 范围表达默认只写本次要做什么；通常不要专门写“不展开范围”“不包含内容”“不做什么”。没有纳入本期的内容，直接不写即可，除非用户明确要求做范围边界、排除项或版本对比。

## 页面与能力类型判断

先判断用户完成任务的主载体，再选择结构。不要只按关键词判断。

| 类型 | 适用场景 | 处理方式 |
| --- | --- | --- |
| 对话 / A2UI 卡片类能力 | 老师通过对话发起任务，系统生成展示卡片、待确认卡片、结果卡片或完成状态卡片 | 卡片只读，不写卡内编辑、选择、确认、提交、刷新等交互；若需要写卡片状态、S1、AI 气泡、卡片空态，读取 `card-design-rules.md`。 |
| 抽屉流程类能力 | 老师通过对话发起复杂任务，右侧半屏抽屉加载业务页面或 A2UI 创建页面，由 AI 预填和推荐，老师校对并确认 | 使用本文件的抽屉流程规则；不套传统卡片状态结构。 |
| 普通页面 / 后台管理类能力 | 老师在页面中配置、管理、查看、导入、上传、启停、权限维护 | 使用普通页面结构，不读取卡片专项规则。 |
| 路由版综合 PRD | 同时包含对话、卡片、抽屉、异步任务、普通管理页和研发路由口径 | 使用综合型 PRD 结构，并按承载方式拆分规则。 |

判断规则：

- 主载体是页面表单、配置流程、资产维护或权限发布时，按普通页面 / 后台管理类能力处理。
- 主载体是对话返回业务卡片时，按对话 / A2UI 卡片类能力处理。
- 主载体是右侧半屏抽屉加载真实业务页面时，按抽屉流程类能力处理。
- 同一需求同时包含管理页和卡片能力时，先拆分对象；管理页部分使用普通页面结构，卡片部分按需读取 `card-design-rules.md`。

## 综合型 PRD 结构

路由版或综合型 PRD 推荐使用以下结构：

```md
## 0. 文档基础信息
### 0.1 基础信息
### 0.2 名词表
### 0.3 优先级处理规则

## 1. 全局交互模型
### 1.1 基本原则
### 1.2 任务承载方式
### 1.3 样板流程
### 1.4 P0 / P1 任务路由矩阵
### 1.5 卡片能力边界
### 1.6 右侧半屏抽屉流程规则
### 1.7 AI 自动操作边界
### 1.8 权限、安全与审计边界
### 1.9 智能推荐规则
### 1.10 异常、中断与恢复

## 2. 入口或页面说明

## 3. 功能需求

## 4. 验收标准

## 5. 修订记录
```

如果用户要求保留已有结构，优先局部补齐，不强行改成以上结构。

## 功能需求标准结构

综合型 PRD 的 `功能需求` 章节应按“模块 -> 功能”组织，不为每个模块临时设计不同结构。

模块层级建议使用：

1. `模块说明`
   - 说明模块负责的业务对象、服务的老师任务、与其他模块的边界。
2. `功能清单`
   - 用表格列出功能、优先级、承载方式和是否展开详细需求。
   - P0 / P1 功能展开详细需求；P2 只保留占位。
3. `单功能需求`
   - 每个 P0 / P1 功能按统一结构展开。

单功能需求标准结构：

标题编号规则：

- 单功能下的子标题需要继承父级功能编号，不在每个功能下重新从 `1.` 开始。
- 例如 `3.2.5 创建日程` 下应写为 `3.2.5.1 基础信息`、`3.2.5.2 用户目标`、`3.2.5.3 所需字段与处理方式`。
- 修改已有 PRD 时，如果某个单功能子标题仍为 `1. 基础信息`、`2. 用户目标`，应按父级功能编号同步改为完整编号。

1. `一句话描述`
   - 写老师在什么场景下，通过什么方式完成什么结果。
2. `基础信息`
   - 包含功能名称、所属模块、优先级、承载方式、触发入口、完成后回流、是否涉及高影响动作、是否需要老师确认。
3. `用户目标`
   - 写老师为什么需要该功能、减少什么操作、最终获得什么结果。
4. `所需字段与处理方式`
   - 用表格写字段、是否必填、处理方式、缺失处理。
   - 字段处理方式只使用两种类型：智能预填、页面填写。
   - 智能预填需要默认包含老师明确表达、老师记忆、历史习惯、偏好、当前课程上下文等可用信息。
   - 智能预填字段的缺失处理按固定顺序写：先使用 Agent 预测值；Agent 无法预测时再使用唯一确定的规则兜底；必填关键字段仍无法确定时触发排疑；选填字段仍无法确定时留空。
   - 默认值规则必须是单一确定规则，不写“最近 / 下一节课”“可使用 A 或 B”这类多选兜底。
   - 字段如果会出现在 A2UI 卡片或抽屉页面中，PRD 需要同步写清默认值来源；卡片上需要展示的字段，默认视为需要预填或兜底规则。
   - 页面填写表示需要老师在页面中选择、上传、输入或配置。
   - 缺失处理只写明确的默认处理方式；如果需要老师在页面操作或没有特殊处理，则留空。
5. `状态机制`
   - 按承载方式判断是否展开。
   - 抽屉流程类、待确认卡片类、异步任务类必须写状态机制。
   - 展示卡片类如只是查询展示，可以简化状态机制。
   - 纯对话类一般不单独写状态机制。
   - P2 占位能力不写状态机制。
6. `提示流转`
   - 写等待时文字和触发时机。
   - 等待文案要自然、短，不写泛泛的“系统处理中”，不承诺尚未完成的结果。
   - 多步骤抽屉任务只在开始时简短说明，不每一步持续讲解。
   - 抽屉流程类能力不设置“页面校验”环节；老师校对、页面必填项和提交失败等规则写入状态机制、抽屉生命周期或空态与异常。
7. `空态与异常`
   - 写无数据、多候选、权限不足、字段缺失、校验失败、提交失败、抽屉中途关闭、异步失败等真实阻断场景。

单功能字段表推荐格式：

| 字段 | 是否必填 | 处理方式 | 缺失处理 |
| --- | --- | --- | --- |

提示流转表推荐格式：

| 场景 | 触发时机 | 等待/提示文案 |
| --- | --- | --- |

## 内部拆解检查点

以下内容只作为写作时的内部检查，不在 PRD 正文中单独设置写作口径或拆解说明章节。

P0 / P1 单功能需要确认：

- 功能名称、使用场景、业务域、优先级。
- 任务承载方式。
- 触发方式与识别条件。
- 老师需要提供的信息。
- 系统可自动获取、推荐或预填的信息。
- 老师可见结果。
- 关键业务规则。
- 异常与兜底。
- 验收标准。

不同承载方式的写作关注点：

- 纯对话类：触发条件、回答边界、无法回答时的追问方式。
- 展示卡片类：展示数据来源、筛选条件、空态和异常态。
- 待确认卡片类：确认对象、确认后影响范围、老师如何通过对话确认或修改，以及旧卡删除 / 新完成卡返回规则。
- 抽屉流程类：抽屉打开条件、内容区自适应、业务页面或 A2UI 创建页面承载、空白抽屉任务卡片、AI 预填规则、老师确认边界和自然语言完成回流。
- 异步任务类：受理条件、执行中反馈、完成回流、失败处理和结果保存。

管理页能力需要确认：

- 对象关系与归属关系。
- 创建、编辑、删除、停用的影响范围。
- 来源权限。
- 必填条件。
- 授权与校验规则。
- 发布、草稿、不可用等生命周期状态。

## 优先级处理规则

- P0 / P1 能力需要写清任务承载方式、关键规则、权限边界和验收标准。
- P2 能力只作为占位保留，写法为：`占位：后续补充……`。
- P2 不展开详细流程、字段、状态机制、专项验收和页面规则。
- 如果 P2 功能已经写得过细，修改时应收敛为占位口径。
- 学生侧能力如出现在教师端 PRD 中，默认视为相关业务占位，除非用户明确要求展开学生端。

## 任务承载方式

任务承载方式固定按以下顺序写：

1. 纯对话类
2. 展示卡片类
3. 待确认卡片类
4. 完成状态卡片类
5. 抽屉流程类
6. 异步任务类

样板流程也按这个顺序写，每类只保留一个典型流程，不重复写多个同类样板。

## P0 / P1 任务路由矩阵

P0 / P1 功能进入研发拆解前，必须能在路由矩阵中查到承载方式、抽屉规则、AI 自动处理范围、老师确认点和完成后返回内容。

推荐字段固定为：

| 任务 | 优先级 | 承载方式 | 是否打开抽屉 | AI 自动处理范围 | 老师必须确认 | 完成后返回 |
| --- | --- | --- | --- | --- | --- | --- |

补充规则：

- P2 能力不进入路由矩阵的详细拆解。
- 对话流卡片只读，不承载卡内编辑、选择、确认按钮、提交按钮、题目刷新、批量导入、大文件上传、长列表筛选、跨页面校对和高影响最终动作。
- 如果老师需要修改卡片信息，通过对话生成新卡片，不在原卡片中局部编辑或刷新。
- 待确认卡片完成后，可以删除旧待确认卡片，并返回新的完成状态卡片，避免对话中保留冗余卡片。
- 抽屉流程中，AI 可以预填和推荐，但发布、导入、删除、取消、开放权限等高影响动作必须由老师确认。
- 多步骤抽屉流程的完成后返回默认写为 `自然语言完成提示`，不写任务完成卡片。

## 卡片能力边界

对话流中的卡片只作为只读展示容器。卡片可以展示数据、展示待确认信息、展示结果或展示完成状态，但不能承载任何卡内交互。

卡片支持生命周期操作：

- 老师需要修改卡片信息时，通过对话提出修改要求，系统生成一张新卡片。
- 待确认卡片完成后，可以删除旧待确认卡片，并返回新的完成状态卡片。
- 完成状态卡片只展示最终结果，不继续承载编辑、提交或刷新。

卡片不承载：

- 卡内编辑
- 卡内选择
- 卡内确认按钮
- 卡内提交按钮
- 卡内删除按钮
- 卡内刷新
- 卡内展开收起
- 复杂多页面流程
- 批量导入
- 大文件上传
- 长列表反复筛选
- 需要后端反复生成或刷新的“换一批”
- 跨页面校对
- 高影响最终动作

原轻量提交类需求应改为：

- 信息足够简单时，使用只读待确认卡片展示目标对象、关键字段和影响范围，由老师通过对话确认或提出修改。
- 需要老师编辑、选择、上传、校对多个字段或处理提交失败时，进入抽屉流程。
- 需要页面化但不依赖既有业务页面时，可以在抽屉内容区使用 A2UI 创建页面承载。

## 抽屉流程规则

抽屉流程适用于复杂、多步骤、需要页面配合的任务，例如创建作业、创建课程、导入学生、发布考试。

需要写清：

- 什么时候打开抽屉。
- 抽屉加载哪个业务页面；如果没有合适的既有业务页面，是否使用 A2UI 创建页面承载。
- 抽屉内容区是否需要按内容区宽度自适应。
- AI 可以自动预填和推荐什么。
- AI 开始时如何用简短自然语言说明已按老师习惯和数据预填，需要老师校对。
- 是否需要空白抽屉任务卡片。
- 老师必须确认什么。
- 中途关闭怎么处理。
- 完成后如何用自然语言告知任务完成。

多步骤抽屉流程规则：

- AI 只在开始时做简短自然说明，表达已按老师习惯和可用数据进行预填，需要老师校对。
- AI 不在每个步骤持续讲解；只有出现阻断、异常、权限不足或老师主动追问时，才补充说明。
- 提示流转中不写“页面校验”环节；页面内校对、必填项提示、提交失败和重试属于业务页面或异常规则。
- 抽屉内容区可以加载已有业务页面，也可以使用 A2UI 创建页面；两者都需要按内容区宽度自适应展示。
- A2UI 创建页面只用于抽屉内容区，不回退成对话流卡片交互。
- 对话区展示空白样式的抽屉任务卡片。
- 抽屉任务卡片不写业务内容、步骤、进度、推荐理由和说明文案。
- 任务未完成时，抽屉任务卡片可点击，用于打开或恢复右侧抽屉查看进度。
- 任务完成后，抽屉任务卡片置为不可点击。
- 任务完成后，AI 用自然语言告知任务完成，不返回任务完成卡片。

抽屉生命周期规则：

- 抽屉打开失败时，对话区提示老师稍后重试或进入原页面处理。
- 页面加载中断时，保留当前对话，不生成完成提示。
- 老师中途关闭抽屉且业务未完成时，保留抽屉任务卡片可点击状态，方便老师继续打开查看进度。
- 抽屉内支持暂存的页面，老师再次发起同类任务时优先恢复暂存内容。
- 抽屉内提交失败时，保留页面已填内容，并提示失败原因和可重试动作。
- 抽屉内业务完成后，对话区用自然语言告知任务完成，并将抽屉任务卡片置为不可点击。

## AI 自动操作边界

- 老师已经明确给出的信息，AI 可以自动填写。
- 不涉及最终提交、发布、删除、导入等高影响动作，且不需要老师校对的页面步骤，AI 可以自动推进到下一步。
- 需要老师校对、存在多个候选、影响学生或课程数据、涉及发布或导入的步骤，AI 只做推荐和预填，由老师自行点击下一步、提交或发布。
- AI 不应在老师未确认的情况下完成高影响最终动作。
- 推荐依据不足时，AI 应说明不确定点，并让老师选择或补充。

## 权限、安全与审计边界

涉及 V5 小管家、Agent、MCP、学生数据、成绩数据、跨课程数据时，必须写权限、安全与审计边界。

至少写清：

- 当前用户能访问什么数据。
- 跨课程、跨班级读取数据时按什么角色权限判断。
- 学生个人信息、成绩、提交记录和学习分析如何最小化展示。
- 发布、导入、删除、取消、开放权限、启用 API、启用外部 MCP 等高影响动作是否必须确认。
- MCP 授权未完成或连接不可用时如何处理。
- server 配置中的敏感信息是否明文展示。
- 关键操作日志记录哪些内容。

## 普通页面 / 后台管理类能力结构

普通页面、管理页、配置页、导入上传页、权限管理页，不强制使用对话卡片结构。除非用户另有要求，使用以下结构：

1. `模块基础信息`
   - 包含 `功能名称`、`功能描述`、`使用场景`、`业务域`、`优先级`、`当前状态`。
2. `用户目标`
   - 写老师要完成的管理动作和最终结果，不写页面控件清单。
3. `核心关系`
   - 写清对象层级、归属关系、依赖关系和数量关系。
4. `关键规则`
   - 写页面上不一定能看出来、但研发和测试必须知道的规则。
   - 字段表只列业务必需字段，不罗列原型中已经能表达的展示字段、按钮、图标和布局。
5. `状态与生命周期`
   - 当对象存在草稿、已发布、停用、校验失败、授权未完成、待重新发布等状态时，写清状态含义、触发条件、可执行操作和影响范围。
6. `权限与边界`
   - 写清不同来源、不同角色、不同状态下能否查看、选择、编辑、删除、启用、停用。
7. `异常与兜底`
   - 只写真实阻断或影响业务正确性的情况。
8. `验收标准`
   - 写可测试的业务规则和边界，不写视觉还原项。

## 智能体与能力管理规则

写 Agent、Skill、MCP、知识库、工作流时，必须明确：

- Agent 是使用入口，Skill / MCP / 知识库 / 工作流是可关联能力。
- Agent 与 Skill / MCP / 知识库 / 工作流可以是多对多关系。
- 关联级操作只影响当前 Agent。
- 资产级新增、编辑、删除、停用会影响所有已关联的 Agent。
- 标准和定制 Skill / MCP 只展示摘要，不支持老师查看详情和编辑。
- 自建 Skill / MCP 支持查看详情、编辑、删除和停用。
- MCP 需要授权时，授权信息通过 server 配置占位承接。
- 扣子 Agent 导入如未设计完整迁移流程，只保留入口占位。

## PRD 修改要求

修改已有 PRD 时：

- 不覆盖原文件，除非用户明确要求。
- 不删除用户已确认的产品口径。
- 发现旧口径和新口径冲突时，以用户最新确认的口径为准。
- 修改一个规则时，检查相关章节是否也需要同步：名词表、全局规则、功能表、路由矩阵、样板流程、验收标准。
- 删除或改写过时内容，不保留“旧版如何”的历史解释，除非用户要求做版本对比。
- 对 P2 能力保持占位，避免误写成本期交付。
- 对普通管理页，不强行套卡片结构。
- 对综合 PRD，不把写作规范、拆解检查点或写作口径写成正文板块。
- 收敛范围时，优先从功能清单、路由矩阵、验收标准中删除未纳入内容；不要新增“本期不展开范围”“不包含内容”章节，除非用户明确要求。

## 避免事项

- 能直接定义时，不写“不是……而是……”。
- 能直接写本期范围时，不写“本期不包含……”“不展开……”“暂不支持……”这类排除性说明；未纳入内容通常直接省略。
- 不添加未经确认的字段名、JSON 路径、枚举名、后端变量名或智能体标识。
- 不把执行手册、实现代码或研发过程写成 PRD 交付物。
- 不添加不能直接支撑当前章节的上下文。
- 不解释“应该如何看这份文档”，除非用户要求。
- 不引入对文档读者无意义的信息，例如 `不展示骨架屏`。
- 不写“当前状态：已按某某要求修改”这类作者日志。

## 推荐改写方式

- 将实现命名改成产品语义。
- 将元说明改成直接定义。
- 将重复段落整理成表格或规则列表。
- 保留产品决策，删除作者评论。
- 写“按钮可用条件”，不写未经确认的前端状态名。
- 写“返回给老师的信息至少包含……”，不写未经确认的接口字段。
- 对复杂能力先补路由矩阵，再补具体功能规则。
- 对卡片类能力，先确认卡片是否只读、修改是否通过对话生成新卡、完成后是否删除旧卡并返回完成状态卡片。
- 对多步骤抽屉能力，先补空白抽屉任务卡片、业务页面或 A2UI 创建页面承载、内容区自适应、可点击 / 不可点击和自然语言完成回流规则。

## 输出验收标准

使用本 Skill 输出 PRD、缺口分析或改写稿前，需要满足：

- 已判断目标属于对话卡片、抽屉流程、普通管理页还是综合 PRD。
- P0 / P1 功能已补齐可研发拆解的信息。
- P2 功能仍保持占位。
- 综合型 PRD 的功能需求已按模块 -> 功能组织，P0 / P1 单功能已包含一句话描述、基础信息、用户目标、所需字段与处理方式、状态机制、提示流转、空态与异常。
- 如果是传统对话 / A2UI 卡片详细能力，已读取并遵循 `card-design-rules.md`。
- 如果是对话流卡片能力，卡片必须只读；没有写卡内编辑、选择、按钮确认、提交、刷新、展开收起或跳转。
- 如果是普通页面 / 后台管理类能力，没有使用 S1/S2、AI 气泡、卡片空态、卡片状态机制等卡片专项结构。
- 如果是待确认卡片，已写明老师通过对话确认或修改、系统可删除旧卡并返回新的完成状态卡片。
- 如果是多步骤抽屉流程，已写空白抽屉任务卡片、业务页面或 A2UI 创建页面承载、内容区自适应、可点击 / 不可点击和自然语言完成提示，且没有写成持续逐步讲解或任务完成卡片。
- 涉及对象状态变化时，已写清状态含义、触发条件、可执行操作和影响范围。
- 涉及学生数据、成绩、MCP、外部服务或高影响动作时，已写权限、安全与审计边界。
- 验收标准能被研发或测试直接验证。
- 输出只包含本次任务需要的正文、缺口或修改说明，不夹带执行过程和作者日志。

## 检查清单

完成前检查：

- 每一段都直接服务于文档目标。
- 标题层级和编号连续；单功能子标题继承父级功能编号，不重新从 `1.` 开始。
- 名词表、功能表、路由矩阵、验收标准之间没有口径冲突。
- 样板流程是否按承载方式顺序排列，且每类只有一个典型流程。
- P0 / P1 是否有路由矩阵或可研发拆解规则。
- 功能需求中的 P0 / P1 单功能是否补齐一句话描述、基础信息、用户目标、字段处理方式、状态机制、提示流转、空态与异常。
- 状态机制是否按承载方式决定展开程度，没有把所有功能强行写成传统卡片状态。
- P2 是否保持占位。
- 卡片边界是否明确只读，并避免卡内编辑、选择、确认按钮、提交按钮、刷新、复杂流程、批量导入、大文件上传、题目刷新和高影响最终动作。
- 待确认卡片完成后，是否写清可以删除旧卡并返回新的完成状态卡片。
- 抽屉流程是否写清打开、业务页面或 A2UI 创建页面承载、内容区自适应、预填、老师确认、空白抽屉任务卡片、关闭、失败和自然语言完成回流。
- 普通页面 / 后台管理页面是否删除了纯页面控件清单。
- 是否写清资产级操作与关联级操作的影响范围。
- 是否写清权限、安全与审计边界。
- 是否存在不必要的“不展开范围”“不包含内容”“不做什么”等排除性段落；如无明确需要，应删除并只保留正向范围。
- 是否存在作者日志、执行说明或智能体操作备注。
